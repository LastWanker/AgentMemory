# V3 计划书

更新时间：2026-02-28

## 0. 目标

在母项目旁边寄生一个新的 `_V3/` 工作区，尽快做出一版干净、清爽、可讲述、适合 demo 展示的记忆检索系统。

这版 V3 的总目标不是“兼容历史”，而是：

- 只保留一个主通道：`reranker`
- 把粗召回明确收敛为 `hybrid`
- 让 cluster 既服务训练，也服务扩召和展示
- 让目录、配置、脚本、运行产物全部隔离
- 为后续 Agent、UI、Java 后端预留清楚边界

## 1. 目录与隔离原则

V3 当前目录骨架：

```text
_V3/
  README.md
  readme/
  src/
  scripts/
  configs/
  runs/
```

隔离规则：

- `_V3/src` 不引用母项目 `src` 作为运行依赖
- `_V3/scripts` 不调用母项目脚本作为默认入口
- `_V3/configs` 自己维护最小配置集
- `_V3/runs` 单独保存日志、指标、样本快照
- 顶层 `data/` 允许共享，但 V3 产物统一进入 `data/V3/...`

## 2. V3 第一版默认架构

默认链路冻结为：

1. 从共享 `data/` 读取原始或母项目已整理的数据
2. 生成 V3 自己的 memory/query/cluster/slot/sample 产物
3. 执行 `hybrid` 粗召回：
   - `dense top 50`
   - `BM25 top 50`
4. 对基础候选做 dedup union
5. 按 cluster 规则执行扩召
6. 对每条 memory 执行 8 槽抽取并生成 8 个槽向量
7. 用 V3 reranker 对扩召后的 list 做精排
   - 继续保留 V2 的 `8x8` 精排思路
   - 但“8 个向量”的来源改为 8 个槽位向量，而不是原样沿用旧 token 选法
   - 若保留 token 备选方案，则按 `top-32 -> MMR 选 8` 做多样性约束
8. 输出 top-k 和记忆命中解释

## 3. V3 第一版明确不做

- 不兼容母项目的 legacy runtime
- 不保留 `simple/tiny` 等过渡态模型家族
- 不做长期并行的 `meta/lexical/semantic/coarse` 多通道融合
- 不先做复杂蒸馏、多模型混训、在线学习
- 不把模型核心直接搬成 Java/Go

## 4. 配置冻结方案

V3 第一版建议只保留 3 个配置文件：

1. `_V3/configs/default.yaml`
2. `_V3/configs/local.example.yaml`
3. `_V3/configs/deepseek.env.example`

第一版默认配置项只保留最关键的：

- 数据路径
- dense encoder
- BM25 参数
- dense/bm25 候选数
- cluster 扩召阈值与上限
- 8 槽抽取配置
- reranker 模型参数
- 训练负例比例
- 当前 query augmentation 开关
- 服务/API 配置

默认方案：

## P3 cluster 扩召与训练样本生成

目标：
- 把 cluster 从“静态标签”升级成“扩召和训练核心”

默认规则建议：
- 触发条件：`matched_count >= 2 and matched_count / cluster_size >= 0.5`
- 扩召后总候选上限：`150`

负例默认：
- `50% near-miss`
- `50% random out-of-cluster`

query augmentation：
- 先沿用当前已有做法
- 第一版不额外新增 `perspective augmentation` 复杂分支

建议新增：
- `_V3/src/.../sampling/cluster_expander.py`
- `_V3/src/.../sampling/build_training_samples.py`
- `_V3/scripts/.../build_training_samples.py`

验收：
- 单条 query 可生成完整训练 list
- list 长度允许动态变化
- 日志能解释扩召前后候选规模

## P4 槽化主链与 reranker V3

目标：
- 继承 V2 的有效思路，但在 `_V3` 里独立实现
- 把槽化从 sidecar 提升为主线路输入

第一版建议：
- 继续采用多向量 / token-level 对齐思路
- 保留 `8x8` 风格输入，但不要直接复制母项目实现
- 默认 8 个槽位固定为：
  - `raw_brief`
  - `event`
  - `intent`
  - `entities`
  - `status`
  - `emotion`
  - `context`
  - `impact`
- 每个槽位单独 embed，构成 8 个输入向量
- `entities` 槽默认把实体列表拼成一行再 embed
- 训练支持动态 list + padding mask
- 槽位抽取使用 LLM IE + 严格 JSON 契约，不另训复杂小模型
- 若后续保留 token 备选方案，则使用 `top-32 -> MMR 选 8`

建议新增：
- `_V3/src/.../slots/extractor.py`
- `_V3/src/.../slots/prompts.py`
- `_V3/src/.../models/reranker.py`
- `_V3/src/.../training/trainer.py`
- `_V3/scripts/.../train_reranker.py`
- `_V3/scripts/.../eval_reranker.py`
- `_V3/scripts/.../build_slots.py`

默认评测指标：
- `CoarseRecall@N`
- `CoarseRankRecall@k`
- `Recall@k`
- `Recall@P`
- `Gain@k`
- `MRR`
- `Top1`

验收：
- 能稳定训练与离线评测
- 能区分粗召回问题和 reranker 增益问题

## 6. 第一版默认技术决策

建议先直接拍成下面这组默认值：

1. hybrid 的关键词召回固定为 `BM25`
2. base coarse candidates 固定为 `100`
3. cluster 扩召后总候选上限固定为 `150`
4. 训练负例固定 `near-miss 50% + random 50%`
5. reranker 第一版保留 `8x8` 多向量对齐主思路
6. 8 个输入向量默认来自 8 槽位文本的 embedding
7. token 备选方案保留 `top-32 -> MMR 选 8`
8. 不先训槽模型，先用 LLM IE 主线路
9. `_chat` 第一版用 Python 落地
