# V4 阶段性计划书_E5迁移阶段

更新时间：2026-03-01

## 0. 阶段目的

V4 不再沿用 V3 当前的 `TF-IDF + char TF-IDF + SVD` dense 路线，而是吸收方案 B / v2 已经踩坑得出的经验，切回：

- `E5 sentence embedding` 负责 coarse
- `E5 sentence slot` 负责 8x8 精排输入
- 全流程优先走缓存、离线、固定入口

目标不是把 V2 原样搬进 V4，而是保留 V4 已经验证过的精简结构，同时把 V2 在编码器、缓存和离线控制上的成熟经验一次性吸收过来。

## 1. 从 v2 吸收的硬经验

以下经验来自现有日志和实现，V4 直接继承，不再重踩：

1. E5 prefix 必须统一
说明：
- `query:` 只能用于 query 编码
- `passage:` 只能用于 memory / slot 编码
- 必须避免双前缀

依据：
- `README/开发日志/开发日志_tinyreranker阶段.md`
- `src/memory_indexer/encoder/hf_sentence.py`
- `src/memory_indexer/encoder/e5_token.py`

2. 默认强离线
说明：
- 不允许自动联网下载模型
- 默认 `hf-local-only + hf-offline`
- 缺模型时给出明确错误，不要静默退化

依据：
- `scripts/memory_indexer/train_reranker.py`
- v2 运行日志里已经明确把联网问题当成踩坑点处理过

3. cache-only fast path 必须保留
说明：
- 当 memory/query 缓存都完整可复用时，跳过 HF/E5 初始化
- 这是 v2 降低等待感和避免重复加载的关键经验

依据：
- `README/开发日志/开发日志_tinyreranker阶段.md`
- `README/开发日志/开发日志_方案B阶段.md`

4. 固定缓存别名比复杂签名更适合用户链路
说明：
- 正式链路使用固定 alias
- smoke 链路使用独立 alias
- 默认不要混用

依据：
- v2 的 `users / users_smoke`
- v2 doctor 对缓存污染的检查

5. 训练、评测、聊天都不应频繁重新加载 E5
说明：
- 进程内复用 encoder
- 优先复用磁盘缓存
- 命中缓存时应直接跳过 encoder 初始化

## 2. V4 当前状态与主要问题

V4 目前是从 V3 复制出的隔离工作区，基础路径已经改到：

- `_V4/...`
- `data/V4/...`

但编码器主线还没有切到 E5，因此当前仍有这些问题：

1. dense coarse 仍不是 transformer 语义编码
2. slot 向量仍不是 E5 向量
3. `data/V4/VectorCacheV4` 目录存在，但主链尚未接入正式缓存协议
4. `_chat` 若直接接 V4，仍继承 V3 那套非 E5 的 dense 行为

## 3. V4 的目标形态

V4 冻结为下面这条主链：

1. `build_dataset`
2. `build_query_slots`
3. `build_slots`
4. `build_e5_cache`
5. `build_hybrid_index`
6. `build_training_samples`
7. `train_reranker`
8. `eval_offline`
9. `_chat` / service

其中最关键的新节点是：

`build_e5_cache`

它负责把：

- memory 原文
- query 原文
- 8 个 slot 文本【批阅：我得提醒你一句，8*8意味着不论是memory还有query都是有8个slot文本的。】

统一编码并写入 `data/V4/VectorCacheV4`。这里的 8 槽明确同时覆盖 memory 和 query，两侧都要有完整的 8 槽文本。

## 4. 编码策略

### 4.1 coarse

coarse 使用 E5 sentence embedding，并继续保留 hybrid 结构。也就是：E5 dense 50 + 关键词 BM25 50 + cluster 连带扩召；E5 只替换掉旧的 `TF-IDF / SVD` 句向量粗召回，不替代关键词通道。

具体约束：

- query coarse：`query: <text>`
- memory coarse：`passage: <text>`
- 全部 L2 normalize
- coarse matrix 离线落盘，不在请求时重算

### 4.2 slot

slot 使用 E5 编码，并作为 V4 对 v2 `top8 token` 方案的正式替代。

这里的设计意图不是“先用 slot 顶一下，以后再回到 token-top8”，而是：

- 用 LLM 先把记忆拆成 8 个有语义的槽
- 再分别对这 8 个槽做 E5 编码
- 直接形成 `query 8 槽 x memory 8 槽` 的 8x8 输入

也就是说，V4 吸收 v2 的是：

- E5 编码器
- `query:/passage:` prefix
- 缓存与离线经验

而不是把 v2 的 token-top8 表达形式继续带进来。

当前阶段冻结为：

- 每个 slot 文本一个 E5 sentence vector
- query 8 槽 x memory 8 槽 直接进当前 8x8 bipartite
- 空槽统一记为零向量并带 mask
- pooling / 对齐时只在有效槽上计算，避免空槽把分数带歪

### 4.3 prefix

V4 一律遵守：

- query 端：`query:`
- memory / slot 端：`passage:`

禁止：

- 自动猜角色
- 重复加前缀
- 部分路径加、部分路径不加

## 5. 缓存设计

### 5.1 目录

V4 正式缓存目录固定为：

- `data/V4/VectorCacheV4/`

推荐文件拆分：

- `memory_sentence_cache_users_v4.jsonl`
- `query_sentence_cache_users_v4.jsonl`
- `memory_slot_cache_users_v4.jsonl`
- `query_slot_cache_users_v4.jsonl`

smoke 单独一套：

- `*_users_smoke_v4.jsonl`

### 5.2 cache key

至少包含：

- `row_id`
- `text`
- `encoder_id`
- `role`
- `slot_field`
- `cache_version`

### 5.3 cache-only fast path

必须提供：

1. memory cache 完整复用检查
2. query cache 完整复用检查
3. 命中后跳过 E5 初始化
4. 误判时自动回退到真实 encoder 初始化

这条是 V4 必做，不是优化项。

## 6. 离线与联网策略

V4 默认策略：

1. `local_files_only = true`
2. `offline = true`
3. 模型不存在时直接报错
4. 不在主链里自动下载 HuggingFace 模型

需要在线下载模型时，只允许显式单独执行一次“模型预热/预下载”入口，不混到训练和评测里。

## 7. 模型加载策略

V4 统一要求：

1. 一个进程里最多初始化一次 sentence encoder
2. 一个进程里最多初始化一次 token encoder
3. 优先走缓存命中再决定是否加载 encoder
4. `_chat` 服务常驻进程内复用 encoder，不允许每轮请求重复初始化

## 8. 与 V3 的边界

V4 不是推翻 V3。

V4 直接继承并保留：

1. `_V4` 的隔离目录结构
2. 当前 `8x8 bipartite` reranker 骨架
3. `cluster` 扩召逻辑
4. `_chat` 并列部署形态

V4 明确替换的只有一件大事：

`dense / slot 编码方式`

也就是：

- 从 `TF-IDF + SVD`
- 切到 `E5 + VectorCacheV4`

## 9. 实施步骤

### P1. 基线修正

目标：

- 所有 `_V4` 默认路径改为 `_V4` / `data/V4`
- `runs`、`models`、`processed`、`exports` 不再误指向 V3

验收：

- `_V4` 所有默认命令不再引用 `_V3` 与 `data/V3`

### P2. 编码器层引入

新增：

- `agentmemory_v4` 或沿用现包名下新增 `encoder/`
- `hf_sentence.py`
- `e5_token.py`
- `hf_runtime.py`

要求：

- 默认模型固定为 `intfloat/multilingual-e5-small`
- 默认开启 prefix
- 默认强离线

### P3. VectorCacheV4 协议

新增：

- `build_e5_cache.py`
- `cache_manifest.json`
- `doctor_cache_v4.py`

要求：

- memory/query 粗向量和 slot 向量全部支持离线落盘
- 支持 `users` 与 `users_smoke`
- 支持 cache-only fast path

### P4. coarse 改造

修改：

- `build_hybrid_index.py`
- `retrieval/dense_index.py`

要求：

- dense matrix 来自 E5 coarse cache
- 不再由 TF-IDF+SVD 直接生成 dense coarse
- BM25 保留为第二路粗召回

### P5. slot 向量改造

修改：

- `build_query_slots.py`
- `build_slots.py`
- `build_hybrid_index.py`
- `training/slot_sequences.py`

要求：

- 8 个 slot 文本统一走 E5 编码
- slot sentence vector 作为 V4 的正式表达，不再把 token-top8 作为回归目标
- 空槽统一零向量 + mask，不能把 8x8 对齐带歪

### P6. 训练与评测接入

修改：

- `build_training_samples.py`
- `train_reranker.py`
- `eval_offline.py`

要求：

- 样本、训练、评测全部从 VectorCacheV4 取向量
- 命中缓存时跳过 encoder 初始化
- run summary 明确记录 encoder、cache alias、offline 状态

### P7. `_chat` 接入

修改：

- `_chat` 的 adapter / service

要求：

- `_chat` 能优先接 V4 retriever
- 服务常驻复用 encoder
- 不在每次请求时重新加载 E5

## 10. 验收标准

V4 E5 迁移完成后，至少满足：

1. 默认命令全指向 `_V4` 与 `data/V4`
2. 默认运行不联网
3. 第二次训练/评测明显快于第一次
4. 命中缓存时日志明确显示跳过 HF/E5 初始化
5. coarse 在模糊 query 与 chat query 上更接近 v2 体感
6. `_chat` 在真实对话里不再出现“每轮像重新热启动”的等待感

## 11. 本阶段不做

先不做下面这些，避免再次臃肿化：

1. 不同时保留多种 dense backend 并长期并行
2. 不把 smoke 和 formal 共用同一 alias
3. 不在训练或评测时自动联网拉模型
4. 不把 v2 的 token-top8 机制重新带回 V4
5. 不在 V4 里复制 v2 所有历史兼容逻辑

## 12. 结论

V4 的正确方向不是继续在 `TF-IDF + SVD` 上加补丁，而是：

- 把 V2 已验证有效的 E5 编码经验一次性吸收
- 但只吸收“必要且干净”的部分
- 保持 V4 结构精简
- 用 `VectorCacheV4 + 强离线 + cache-only fast path` 把体验和效率一起稳住
