## 0. 这次 V3 的边界

先把边界写死，否则 V3 很容易再次长成母项目那种“历史层叠体”。

- V3 以根目录下 `_V3/` 为新的隔离点。
- V3 不复用母项目的 `src/`、`scripts/`、`configs/`、`runs/`。
- V3 只和母项目共享顶层 `data/` 的物理存储，但后续应使用 V3 自己的子命名空间，例如 `data/V3/...`。
- 本轮先讨论，不写实现；先把架构、默认值、术语和裁剪面定下来。
- V3 只研究一个主通道：`reranker`。
- V3 的在线链路先简化成：`hybrid 粗召回 -> cluster 连带扩召 -> reranker 精排 -> 展示/写回`。

## 1. 纵观母项目后，V3 应该继承什么

从当前仓库结构和文档看，母项目已经沉淀了几件真正有价值的东西，V3 不该丢：

- `cluster` 作为正例组织单位，这一点是主线，不是附属功能。
- `query augmentation` 是成立的，至少对聊天记忆场景成立。
- 粗召回、精排、评测指标已经有清晰语义边界：
  - `P`：positives
  - `C`：coarse recall candidates
  - `R`：final reranked results
- 训练和评测都应该围绕同一个候选池定义，不再让 query 行内旧字段偷偷参与候选构建。
- V2 已经证明：真正的瓶颈不只是“模型大小”，而是“数据协议、候选协议、负例协议、融合协议”。

## 2. V3 应该明确砍掉什么

V3 如果要干净，下面这些不该再带进来：

- 不再保留 `simple/tiny/legacy/过渡态` 多入口心智负担。
- 不再保留 `S-only` 这种历史命名。V3 里它就是 `reranker` 主通道，不需要历史包袱名。
- 不再把 `meta + lexical + semantic + coarse` 做成一套长期并行融合体系。
- lexical 在 V3 里只承担粗召回的一半，不承担最终融合通道身份。
- 不再让运行脚本和默认值横跨多个年代的目录并存。
- 不再允许“正式结果”和“烟测结果”写入同一套 V3 运行目录或缓存命名空间。

## 3. 对你列出的 11 点逐条讨论

### 3.1 配置数量

你的方向是对的。

V3 第一版应尽量把配置压到最少，建议最多三层：

- `base`：正式默认值
- `local`：本机覆盖
- `env.example`：API/密钥示例

不建议一开始就保留多套 profile。先把默认正式链路跑通，再考虑 smoke profile。

### 3.2 粗召回是否应该叫 hybrid

是，`hybrid` 是业内最常见、也最容易沟通的名字。

V3 里我建议把粗召回写成：

1. `dense top 50`
2. `keyword top 50`
3. `dedup union`
4. `cluster expansion`
5. 交给 reranker

这里的“关键词召回”，第一版最常用、最稳、最省工程成本的实现就是 `BM25`。如果后面你追求更强的 sparse 路线，再考虑 `SPLADE` 一类学习型稀疏检索；但 V3 第一版不应该先上这个复杂度。

行业上也普遍把“向量检索 + BM25”直接称为 hybrid search，常见融合方式是并行检索后做 rank fusion 或 score fusion。对于 V3 第一版，我更建议固定数量并做并集，不要一开始就把融合规则做复杂。

### 3.3 粗召回候选至少 100，且执行连带，训练是否允许动态长度

允许，技术上完全可行。

但要把两个长度分开记：

- `base_candidates = 100`
- `expanded_candidates = dynamic`

否则你后面会把“粗召回命中率问题”和“cluster 连带膨胀问题”重新混在一起。

更具体地说：

- 检索器固定先拿 `50 dense + 50 BM25`
- 去重后得到基础候选集
- 再执行 cluster 连带扩召
- reranker 的 list 长度对 batch 来说可以是动态的，只要 collate 做 padding + mask

训练层面没有原则性问题，但工程上要防两件事：

- 大 cluster 会把 list 长度拉爆
- 评测时不同 query 的候选规模差异过大，指标解释会变脏

所以 V3 里建议增加一个扩召上限，例如：

- 只有当某 cluster 在基础候选里命中比例 `>= 50%` 且命中条数 `>= 2` 时才扩召
- 扩召后的总候选数设上限，例如 `160` 或 `200`【批阅：按数学上计算，粗估测最多也就是150左右的上限就够了。】

### 3.4 “-1 层拿出来更丰富”这件事，业内是否常见

我不建议把这个当成 V3 第一版的主方向。

更准确地说：

- “显式拿倒数第二层做检索/重排表示”不是业界默认做法
- 更常见的是直接使用已经为检索设计好的单向量或多向量模型
- 如果要保留 token 级细节，通常会走 late interaction / multi-vector 路线，而不是手工执着于 `-1` 层本身
【批阅：保留方案 B也就是v2 的 8×8 精排，但是在“选 8 个向量”这一步加 MMR 多样性约束（top-32 → 选 8）】
你担心的另一个点是成立的：如果仍然只取 `top-8` token/向量，它们确实可能出现两种风险：

- 语义重复，高度同质化
- 滑向表层关键词，对细节覆盖并不均衡

所以问题的关键往往不是“是不是 -1 层”，而是：

- 这 8 个向量怎么选
- 有没有去重/多样性约束
- reranker 怎么消费这些 token 级信息

如果 V3 还沿用 `8x8` 风格，我建议优先研究“选什么 token 更稳”，而不是先研究“取哪一层更玄学”。

- 不建议从 `top-8` token 里硬解释出“这个 token 属于情绪/事件/对象”
- 更建议单独做一个显式的 slot extraction【批阅：×】

第一版可选的常见槽位，我建议是：【批阅：我改动了你的设定。】

- 后续 Java/UI 展示更容易做

### 3.6 cluster 聚类思想延续

应当延续，而且建议继续作为 V3 的一等公民。

V3 的 cluster 不只是训练标签，还应承担三个职责：

- 正例组织单元
- coarse recall 的连带扩召单元
- UI 里的“话题/事件”展示单元

### 3.7 负例采集：临近但错 50%，簇外随机 50%，而且不希望太 hard

这个设计是合理的，而且比“全靠 hard negative”更适合先做一版可控 demo。

我建议把它命名得更清楚一点：

- `near_miss_negative = 50%`
- `random_out_cluster_negative = 50%`

这里的 near-miss 不是 hardest negative，而是“语义相近但 cluster 不同”的温和难负例。这个比纯 ranked/hard 更符合你现在的目标。

### 3.8 query 增广保留

保留，没有问题。

但 V3 里建议明确区分两类：

- `rewrite augmentation`：同义改写、口语化、短问法
- `perspective augmentation`：围绕 cluster 从事件、对象、情绪、结果等角度发问【批阅：暂时不需要，可以用现在的】

第一类更稳，第二类更有价值。

### 3.9 粗召回的 cluster 连带规则

你的 `if 同 cluster 被召回 50% 及以上，就把整个 cluster 放进粗召回` 这个思路是成立的，但需要补两个保险丝：

1. 不能只看比例，还要看绝对命中数  
否则 cluster 只有 2 条时，命中 1 条就触发，太松。

2. 不能无限制扩召  
否则某些大 cluster 会把候选池挤爆。

我建议 V3 默认规则写成：

- 触发条件：`matched_count >= 2 and matched_count / cluster_size >= 0.5`
- 扩召范围：整个 cluster
- 扩召上限：扩完总候选不超过 `200`【150即可。】

### 3.10 reranker 沿用 V2，训练时动态 list = |cluster|，8x8 在槽化后是否更好

我的判断：

- `V2 的 bipartite/8x8` 思路可以继承
- 但 V3 不应“原封不动复制 V2 代码”，而应只继承思想和接口形态
- 槽化可能有帮助，但更适合作为 side feature，不适合第一版就把主模型重写成“槽对槽对齐网络”

换句话说，V3 第一版最稳的做法是：

- 主 reranker 继续走 token-level / multi-vector 对齐【直接走槽化】
- slot 信息先做附加输入或解释层【直接走槽化】
- 动态 list 长度按 query 的 expanded candidate size 走，训练时用 mask【批阅：语义模糊！训练时候的list = |cluster|，如此动态。】

### 3.11 没有 simple、没有 tiny、没有任何过渡态；只有 reranker 通道

我支持。

但为了工程可维护，还是建议保留两个概念上的 baseline，不一定对外暴露成正式通道：

- `dense only coarse`
- `hybrid coarse without reranker`

它们不是产品通道，而是评测基线。否则你后面很难区分“粗召回没做好”还是“reranker 没带来增益”。

### 3.12 你可能忘记但实际必须补的东西

我认为至少还有这些：

- `data contract`：V3 的 memory/query/cluster/sample 字段要重新定一遍【支持】
- `artifact namespace`：共享 `data/` 时，V3 的产物写到哪里【产物写到data/V3里面】
- `evaluation split`：训练/验证/演示集如何切【训练尽量不切，验证或者演示依旧随机1000个。】
- `cluster leakage`：跨会话或跨时间切分时如何避免泄漏
- `candidate explosion control`：cluster 扩召后的上限和日志记录
- `dedup/canonical mem_id`：V3 里 ID 规则要不要重新定义【合适即可。】
- `explainability output`：一次命中了哪些片段、为什么命中，要不要结构化落盘
- `serving contract`：未来 Java 后端和 Python 检索核之间如何通信【暂定。】

### 3.13 类 LoRA 方法

我不建议在 V3 第一阶段就把 LoRA 作为主线。【可以。】

更务实的顺序是：

1. 先把用户反馈写成训练样本和反馈日志
2. 先验证反馈是否真的稳定提升 reranker
3. 再考虑是否用 LoRA/adapter 做低成本增量训练

如果后面真要上，我建议优先作用在 reranker，而不是先碰粗召回编码器。前者风险更可控，也更贴近你的单通道目标。

### 3.14 Agent、skills、UI、本地部署、Java 后端

这些都适合做 demo，但建议分层，不要和检索核心绑死。

我的建议是：

- 检索/训练核心仍放在 Python
- Agent 编排可以先放 Python
- Java 后端做服务壳、会话管理、接口编排、前后端交互
- UI 先做简单聊天界面，并允许展开本次命中的记忆片段和 cluster

这样更现实。原因很简单：

- 你的检索训练资产本来就在 Python 生态
- Java 适合做工程壳和展示层
- 真要做求职 demo，清晰的服务边界比“所有东西都塞到一种语言里”更有说服力

如果你坚持第一版后端就直接 Java，我会建议采用“双进程”：

- Java：API/UI/调度
- Python：检索与模型服务

不要试图在第一版把模型训练和检索核心原地搬成 Java。

## 4. 我给 V3 的默认主方案

如果现在就要尽快做出第一版，我建议直接冻结成下面这个版本：

1. 目录隔离：`_V3/` 独立，母项目仅作参考，不复用代码。
2. 数据共享：仅共享顶层 `data/`，但 V3 产物统一写入 `data/V3/`。【合理。】
3. 粗召回：`hybrid = dense top 50 + BM25 top 50`。
4. 连带扩召：基于 cluster 的规则式扩召，扩后上限 `200`。
5. query 增广：保留，先规则/模板优先，LLM 作为补充。
6. 负例：`50% near-miss + 50% random out-of-cluster`。
7. reranker：继承 V2 的多向量对齐思想，动态 list + mask。
8. 槽化：第一版只做 sidecar，不改主模型结构。【主线路。】
9. 评测：固定看 `Recall@k / Recall@P / CoarseRecall@N / CoarseRankRecall@k / Gain@k`。
10. demo：先做“可对话 + 可展开回忆证据”的简单 UI。

## 5. 这次立项后，我最想反问你的取舍题

3. V3 第一版的 query augmentation，你要不要保留 LLM 生成，还是先只做规则/模板？【依旧使用LLM生成。】
4. slot 你是想先做：
   - 纯展示解释
   - 还是直接进 reranker 特征？【主线路】
5. Java 后端是第一版必须落地，还是第二阶段再接？【日后再说】
6. `data` 共享时，你是否接受 V3 统一写到 `data/V3/`，而不是继续写母项目现有文件名？【是的】
7. 你对 cluster 扩召的上限更倾向：
   - 强控上限，保证解释干净【设计一个顺序优先让大的聚类先补全，超过150则停止】
   - 还是宁可召回更宽，允许候选变大？
