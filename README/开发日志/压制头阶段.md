# 压制头阶段

【维护规则】本日志记录“记忆偏好/抑制”阶段的决策与实现，后续以追加为主。

## 1. 阶段定位
- 背景：联想通路可用后，召回量增加，开始需要“用户偏好抑制”能力。
- 目标：让系统在不破坏主召回链路的前提下，对“无关/遗忘”记忆做轻量降权。
- 原则：开关可热切、默认可 no-op、不开启时不影响原链路、开启后只“压一两个”。

## 3. 决策主线（讨论->冻结）
1. 模型路线：
   - V1 采用 Plain MLP（组合特征 + 轻量网络），先跑通数据链路与在线接线。
   - Oreo（中间 memory layer）保留为升级方案，不阻塞 V1。
2. seed 与通道关系：
   - 不改 coarse/association 主召回逻辑，suppressor 只做后处理扣分。
3. 反馈类型：
   - `unrelated` 与 `toforget` 共用一个模型，`feedback_type` 作为特征输入。
4. 样本观：
   - 保留随机负样本（用于“抚平”无关组合干扰），不只做难负例。
5. 上线方式：
   - 阈值 + 惩罚系数，优先稳定；复杂上下文门控暂后置。

## 4. 反馈数据链路落地
- UI 在每张记忆卡（coarse/association 两路）右侧新增按钮：
  - `无关`
  - `遗忘`
- 点击后行为：
  - 当前卡按钮变灰（同 query 下该 memory 不可重复标注）。
  - 即刻增量写入：`_V5/chat/data/feedback/feedback_events.jsonl`
- 事件内容包含：
  - `session_id`, `query_text`, `memory_id`, `feedback_type`, `lane`
  - `candidate_refs`（含 CandidateCache）
  - `effective_candidate_refs`（自动扣除本 query 已选 self 与本次 self）
  - `query_self_memory_ids_before/after`
- 设计结果：
  - 支持同 query 多个 self（训练样本构造时可用）。
  - 支持离线回放 conversation 旧账生成数据。

## 5. Chat 链路改造（用户体验与对比测试）
- 新增“仅召回模式”开关：
  - 打开后不等待 LLM，助手回合固定“记忆已召回”。
- 新增“记忆偏好”开关（SUPPRESSOR 热开关）：
  - 与“仅召回模式”并列，便于 A/B 对比 top5 变化。
- 请求拆分为两段：
  - `/api/chat/retrieve`：先回召回结果并落盘到会话记录。
  - `/api/chat/respond`：再异步拿 LLM 回复并追加到记录。
- 效果：
  - 正常模式下也能“先看到召回，再看到 LLM”，观感明显改善。

## 7. 指标讨论与阶段性问题
- 用户明确目标分两组：
  1. 反馈 query 集（~200）：
     - 目标是“对应被反馈 memory 在 top5 消失或明显后移”，不是只看任意变化。
  2. 随机 query 集（~2000）：
     - 目标是低扰动，尤其 top1/top5 变化要小。
     - 期望 `dual_changed_count` 约 10%~20%，而不是几乎全变。
- 现象：
  - 当前 suppressor 会对随机 query 产生偏高扰动，是负信号。
- 当前共识：
  - 先做模型调优 + 指标升级 + 门控降噪。
  - 复杂“上下文门控”后置，不作为当前阻塞项。

## 8. 踩坑记录
- 首轮按钮“看不见”问题：前端缓存/版本号导致旧静态资源未刷新。
- 反馈链路初期缺“candidate 扣除”逻辑，后补 `effective_candidate_refs`。
- 仅端到端分数变化不足以解释效果，必须拆分“目标 query 集”和“随机 query 集”双 cohort。
- 过早追求复杂结构风险高：最终先稳 Plain，再保留 Oreo 接口。

## 9. 阶段结论
- 压制头已经从“提纲”进入“可训练、可接线、可热开关、可对比评估”状态。
- 数据链路（反馈事件）已打通，这是后续调优最关键资产。
- 下一阶段重点不是继续堆功能，而是把“定向压制有效 + 随机扰动可控”做扎实。
