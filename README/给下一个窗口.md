# 给下一个窗口（交接）

更新时间：2026-02-27

## 1) 先说当前现状（重点版）

### A. 评测语义与指标口径已统一
- `P` = positives（正例集合，通常是 cluster 内 mem_id）。
- `C` = coarse recall top_n（检索候选池）。
- `R` = 最终排序结果 top_k。
- 评测侧不再使用 query 行内 `candidates/hard_negatives` 作为候选来源。
- 指标当前看三条：`RetrievalRecall@N`、`Recall@k`、`Recall@P`。

### B. 关键性能改造已落地（检索链路）
- `CoarseIndex.search` 已是 CPU 矩阵化 + top-n 选择优化（不再是旧的 Python 全量点积+全排序）。
- 评测已支持细粒度计时输出：`coarse/lexical/scorer/routing` 四段。
- learned scorer 支持 batch 打分；cache 全命中可跳过 HF encoder 初始化。
- quick 入口可用：`scripts/memory_indexer/runtime/eval_followup_plus_chat_quick.py`。

### C. 现在真正卡点从“检索算子”转向“训练协议”
- 当前主要问题不是再问“谁是大头”，而是模型训练协议与目标未完全对齐业务设计。
- TinyReranker 仍是统计池化后 `36 -> 32 -> 1` 的小模型，容量偏保守。
- `mix(auto)` 仍是启发式通道权重，不是“coarse 主路 + lexical 补充”机制。

### D. 缓存现状（本轮核查）
- `data/VectorCache/memory_cache_users.jsonl` 存在，约 400MB+，这是要保留的主缓存。
- `data/VectorCache/eval_cache_users.jsonl` 当前不存在，需要按 full eval 重建。
- `--cache-alias users --no-cache-signature` 仍是正式路径，但必须严格区分 smoke alias。

## 2) 下个窗口优先级（老带新执行顺序）

1. P0：先做缓存地基修复，不碰训练代码。
- 只重建 `eval_cache_users.jsonl`（full eval）。
- 明确保留 `memory_cache_users.jsonl`，不做重建。
- 固化 smoke alias：`users_smoke`，禁止烟测写 `users`。

2. P1：训练协议对齐业务目标。
- 训练样本协议落地为：`query = memory 原文 + LLM 增广 query`。
- positives = 同 cluster；negatives = cluster 外随机（先做第一版）。
- listwise 由固定 `list_size` 改为按 cluster/positives 动态长度。
- 默认 `max-queries=0`（全量），runtime 不再默认 2000 截断。

3. P2：模型升级（在协议对齐后做）。
- 保留 tiny 路线但提升表达力（不再只停留在极简两层统计头）。
- 评估时固定看 `CoarseRankRecall@k / Recall@k / Gain@k` 三联指标，判断模型净贡献。

## 3) 必保留的老经验（不要丢）

1. 终端规则：统一 `cmd /c ...`，不要混用 PowerShell 语法。
2. `cmd` + `rg` 防坑：优先 `rg -n -e p1 -e p2 path`，别在一条命令里堆复杂 `|`。
3. 防卡规则：禁止无范围 `git diff`（尤其大 JSONL/权重）；先 `git status --short` 再定点查看。
4. 缓存规则：`users` 是正式缓存，`users_smoke` 是烟测缓存，严禁串用。
5. 长任务中断恢复：先看 `run_dir`、缓存文件、`git status`，再续跑，不要盲接上条命令。
