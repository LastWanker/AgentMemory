# 问题报告：模型副作用与缓存重建计划（2026-02-27）

> 范围：本报告基于代码与当前目录状态的只读核查，不做烟测重跑，不覆盖任何用户缓存文件。

## 1. 已验证事实（先对齐）

1. 当前缓存目录状态：
- `data/VectorCache/memory_cache_users.jsonl` 存在，大小 `420,380,671` bytes（约 400MB+）。
- `data/VectorCache/eval_cache_users.jsonl` 当前不存在。

2. 当前数据规模（Processed）：
- `data/Processed/eval_followup_plus_chat.jsonl` 共 `4806` 行。
- `data/Processed/memory_followup_plus_chat.jsonl` 共 `3728` 行。

3. 你提到的“eval_cache_users 被覆盖”在机制上是可能且高概率的：
- `eval_router` 在 `--cache-alias users --no-cache-signature` 下，会直接使用固定文件名 `eval_cache_users.jsonl`。
- 一旦本次运行的 `len(queries)` 与缓存行数不一致，会重写该文件。
- `eval_followup_plus_chat_quick.py` 默认 `--max-eval-queries 1000`，且默认 `head` 采样，确实会把缓存变成“前 N 条”风格。

4. 当前 7.9KB 权重并不等于“坏文件”：
- `data/ModelWeights/pairwise_reranker.pt` 当前大小 `7904` bytes。
- 现模型是 Tiny MLP（`36 -> 32 -> 1`），参数量很小，文件小是结构决定，不代表损坏。

## 2. 你关心的 5 个问题（逐条结论）

1. “两层 + 36 维统计特征是否太弱”
- 结论：你的担心合理。当前模型确实容量偏小，且把 `8x8` 相似度矩阵压成统计量后丢了结构信息。
- 代码证据：`src/memory_indexer/learned_scorer.py` 的 `TinyReranker.extract_features()` 先统计池化，再 `Linear(36,32)->Linear(32,1)`。

2. “是否误用了旧权重/旧训练设定”
- 结论：高风险成立。当前 quick 评测默认直接吃 `data/ModelWeights/pairwise_reranker.pt`，而训练入口历史上存在 `max-queries` 截断路径，且训练样本来源完全依赖 eval 文件当前内容。
- 关键点：`runtime/train_eval_followup_plus_chat.py` 默认 `--max-queries 2000`；`train_reranker.py` 实际会按该值截断训练 query。

3. “mix 是否按预期工作（lex 应该是补充）”
- 结论：当前 `mix(auto)` 不是“补充式”机制，而是通道强度启发式加权；这会出现你说的偏差。
- 代码证据：`Retriever._compute_channel_weights()` 对每通道做 min-max 后取均值，再归一化；不是“coarse 主路 + lexical 仅补召回”。

4. “为什么 loss 降了，线上 Recall@5 反而下降”
- 结论：并不矛盾。当前训练目标与线上指标存在错位：
- listwise 使用固定 `list_size` 截断（默认 50），不是按 cluster 动态长度。
- 训练 negatives 主要来自候选池/随机回填，和你想要的“聚类外稳定难负例”定义并未完全对齐。
- 优化器可把训练 loss 做低，但未必提升 `Recall@5`。

5. “-1 层 + 8 向量是否没用”
- 结论：不是完全没用，但信息压缩很强。
- 当前流程是 `Vectorizer(token_pool_topk, k=8)`，再进入固定 `8x8` 相似度；这对提速友好，但会损失细粒度语义关系。

## 3. 谁是大头（基于你最近日志）

从你给的近期指标看（`timing/query_ms coarse=2.71 | lexical=7.21 | scorer=33.14 | routing=0.20 | total=43.29`）：

1. 单 query 内部大头是 `scorer`（约 76%）。
2. `coarse` 已不是第一瓶颈（约 6%），矩阵化后收益已经兑现。
3. 但全流程总耗时仍有大量“检索外”成本（索引构建/缓存读取/初始化等），所以 `scoring_elapsed` 明显小于 `total_elapsed` 是正常现象。

结论：需要拆解，而且现在四段计时已经足够支撑下一刀，不是盲打。

## 4. 当前代码与目标设计的偏差清单

1. 训练 query 来源：
- 现在是直接读取 eval 文件中的 `query_text`。
- 还没有在训练入口内强制实现“query = memory 本体 + LLM 增广 query”的采样策略。

2. listwise 长度：
- 现在是固定 `--list-size`（默认 50），不是“按 cluster 正例规模动态”。

3. 训练候选池：
- `train_reranker.py` 已强制 `candidate_mode=coarse`，这点是对齐你主赛道诉求的。
- 但负例构造仍偏通用，不是你描述的“按聚类结构明确约束”的版本。

4. mix 策略：
- 目前是启发式权重融合，不是你设想的“Lex 主要做 coarse 补充”机制。

## 5. 建议执行计划（先讨论后动手）

### P0（先修数据地基）
1. 重建 `eval_cache_users.jsonl`，严格按 full eval（`4806`）生成。
2. 保留 `memory_cache_users.jsonl` 不动，不做重建。
3. 固化“烟测专用 alias”：`users_smoke`，与正式 `users` 彻底隔离。

### P1（再修训练协议）
1. 明确并落地训练样本协议：
- query = memory 原文 + LLM 生成 query（可多条）。
- positives = 同 cluster。
- negatives = cluster 外随机（第一版）。
2. listwise 改为动态长度（以 cluster/positives 规模为主），不再固定 50。
3. 默认 `max-queries=0`（全量），runtime 包装脚本不再默认 2000 截断。

### P2（模型升级，不走保守路线）
1. 保留 tiny 路线但提升表达能力（例如多层 MLP 或轻量 cross-feature 头）。
2. 评估目标固定围绕：
- `CoarseRankRecall@k`（粗排基线）
- `Recall@k`（重排后）
- `Gain@k = Recall@k - CoarseRankRecall@k`（模型净贡献）

## 6. 防覆盖约束（本项目必须遵守）

1. 正式跑：`--cache-alias users`。
2. 烟测跑：`--cache-alias users_smoke`（必须）。
3. 烟测禁止写正式文件：
- 禁止在 `users` alias 下配合小样本 `--max-eval-queries` 运行。
4. 任何“重建缓存”动作，先打印目标路径并二次确认 alias。

## 7. 结论（给当前决策）

1. 你的判断是对的：当前问题不只是“模型太小”，而是“数据协议 + 训练目标 + 融合策略”一起偏了。
2. 先做缓存地基修复和训练协议对齐，再谈模型结构升级，收益会更稳。
3. 在不碰数据库前提下，现阶段最该做的是“训练流程重建”，不是再打 retrieval 侧小补丁。
