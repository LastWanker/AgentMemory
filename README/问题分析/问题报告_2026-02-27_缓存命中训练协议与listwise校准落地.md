## 1. 本次执行结论（按任务项对齐）

### 1.1 训练缓存命中逻辑（已执行）
- 已在 `scripts/memory_indexer/train_reranker.py` 增加与评测同类的缓存命中短路：
  - query 缓存命中 + memory 缓存完全可复用时，跳过 HF/E5 初始化；
  - 日志明确输出：`[train] query/memory 缓存完整命中，跳过 HF/E5 编码器初始化。`
- 结论：在缓存完备时继续逐条 `encode_query(...)` 的旧行为不合理，当前已改为“缓存优先、必要时才加载编码器”。

### 1.2 “已有粗召回候选”与负例来源（已执行 + 已讨论）
- “已有粗召回候选”含义：
  - 训练样本先走粗召回 `top_n` 得到候选池，再在候选池内划分正负例并训练 reranker。
  - 是的，这代表粗召回参与训练样本构建（不是参与反向传播参数更新）。
- 负例策略已改回更合理版本：
  - 优先使用“粗召回候选中非正例”的 memory 作为负例；
  - 仅在候选池中无负例时，才从全库（排除正例）随机回填。
- 讨论结论：
  - 相比“cluster 外纯随机负例”，粗召回非正例更贴近线上推理分布，通常更有训练价值；
  - 使用粗召回主要是为了“分布对齐 + 计算可控”，不只是节省开销。

### 1.3 Recall@P / CoarseRankRecall@P（已执行）
- `Recall@P` 已核对并保持为目标定义：
  - `P = |cluster positives|`
  - 在 reranker 排序下取前 `P`，计算 `|Top_P ∩ Positives| / |Positives|`。
- 新增 `CoarseRankRecall@P`：
  - 在粗召回排序下取前 `P`，计算同口径比例；
  - 已接入控制台输出、`eval.metrics.json` 与 bootstrap 汇总。

### 1.4 Listwise “硬约束”近似（已执行）
- 已在 `train_reranker.py` 增加 listwise hard constraint（默认开启）：
  - 新参数：`--listwise-hard-constraint/--no-listwise-hard-constraint`、`--listwise-hard-weight`；
  - 约束形式：对 `neg > pos` 的分数关系加惩罚（softplus），近似推动 top-|P| 区间正例优先。
- 影响讨论：
  - 对小 cluster（2~3）通常更利于 `Recall@P`；
  - 对较大 cluster，可能与 `Recall@k`（固定小 k）存在张力，需要看实测平衡。

## 3. 关键问答（本次讨论结论）

### 3.1 Recall@P 常小于 Recall@k 是否合理？
- 合理。多数样本 `|P| <= k` 时，前 `k` 比前 `P` 预算更宽，召回不低于前 `P` 是正常现象。

### 3.2 Gain@k 仍为负，是否说明小模型成为瓶颈？
- 是高概率瓶颈之一，但不是唯一原因。
- 当前 reranker 仍是轻量统计特征 + 小 MLP，表达上限偏低；同时训练目标、负例难度和候选分布也共同影响 Gain。

### 3.3 `runs/rt/p1_pairwise_retrain_20260227/train.log.txt` 是否是当前权重来源？
- 对 `data/ModelWeights/pairwise_reranker.pt`：是来源 run（pairwise）。
- 对当前 quick 默认评测：不是主来源（quick 已默认改为 listwise 权重）。

### 3.4 pairwise 第 1 轮 loss 约 0.22 是否说明接着旧模型训练？
- 不构成该结论。脚本中未发现从旧 `pairwise_reranker.pt` 自动加载继续训练的逻辑。
- 0.22 起点更可能来自当前损失定义、样本难度与负例分布变化，不必等同“断点续训”。
