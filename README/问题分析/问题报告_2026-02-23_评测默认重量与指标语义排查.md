### 术语统一（本次讨论定稿）

为避免“候选”与“正例”混用，本报告统一如下：
- `P (positives)`：真正例集合（通常是同 cluster 的全部正例），大小 `|P|`。
- `C (candidates)`：检索候选池（retriever 从全局索引召回 top_n），大小 `|C|`。
- `R (ranked results)`：最终展示结果（对候选池排序后取 top_k）。

### 1）`top_n=20/top_k=5` 与 Recall 定义的关系

结论：
- 当前实现是标准 `Recall@k`：
  - `Recall@k = |Top_k ∩ P| / |P|`
  - 代码位于 `scripts/memory_indexer/eval_router.py:119-126`。
- 因此当某 query 的 `|P| > k` 时，`Recall@k` 上限是 `k/|P|`，理论上不可能到 1。
- 该行为在“固定展示预算（如 top_5）”场景下是正常语义，不是计算错误。
- 若需要“排序阶段可达 1”的指标，可新增：
  - `Recall@P = |Top_{|P|} ∩ P| / |P|`
  - 说明：当 `|C ∩ P| < |P|` 时，`Recall@P` 上限受召回池覆盖限制，不一定能到 1。

代码证据：
- `scripts/memory_indexer/eval_router.py:119-126`

补充：
- “cluster 大会被固定 top_k 天然压低”成立（例如固定 `k=5`）。

### 3）`baseline(auto)` 到底是什么

结论：
- `baseline(auto)` 不是“无权重”，而是“通道权重自动估计”。
- 在 ablation 中它对应 `fixed_channel_weights=None`，Router 会按该 query 当前候选的特征强度自动归一化出通道权重。

代码证据：
- `scripts/memory_indexer/eval_router.py:769-770`
- `src/memory_indexer/retriever.py:99-134`

补充：
- 这不是学习出来的智能策略，是启发式自动权重。

### 6）`candidate_mode=coarse` 会不会没用上同聚类候选

结论：
- 目前评测候选池 `C` 来自全局索引（coarse/lexical/union），并不会自动注入“同聚类候选”。

  - 训练脚本可优先吃样本内 `candidates`；
  - 评测脚本仍走全局召回构造 `C`。

代码证据：

  - `src/memory_indexer/retriever.py:379-412`

## 五、本次讨论新增结论（指标语义定稿）

### 1）建议并行输出的三类指标

1. `RetrievalRecall@N = |C ∩ P| / |P|`
   - 含义：检索召回层是否把正例带进候选池（top_n 能力）。
2. `Recall@5 = |Top_5 ∩ P| / |P|`
   - 含义：固定产品预算（只展示 5 条）时的覆盖能力。
3. `Recall@P = |Top_{|P|} ∩ P| / |P|`
   - 含义：在“预算=正例规模”时，排序层把正例推上来的能力。

### 2）三指标联动诊断逻辑

1. `RetrievalRecall@N` 低：召回层问题（top_n / candidate_mode / 粗召回策略）。
2. `RetrievalRecall@N` 高但 `Recall@P` 低：排序层问题（候选内正负区分弱）。
3. `Recall@P` 高但 `Recall@5` 低：预算问题（cluster 大、展示位小）。

### 3）对问题 1 与问题 6 的落地建议

1. 问题 1：保留现有 `Recall@k`，新增 `Recall@P`，避免“固定 k 无法满分”的语义误解。
2. 问题 6：评测与训练都不再使用 query 内 `candidates` 字段；候选池统一定义为“粗召回 top_n（句向量）”。
