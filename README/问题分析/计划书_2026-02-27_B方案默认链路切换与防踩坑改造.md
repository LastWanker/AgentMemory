# 计划书：B 方案默认链路切换与防踩坑改造（2026-02-27）

> 目标：在不再“疯狂投入性能优化时间”的前提下，把 reranker 主链路从 tiny 统计头升级到显式对齐模型，并把新链路设为默认入口。  
> 重点：先锁数据链路正确性与可复现，再做模型改造；不再让缓存、采样、脚本默认值反复污染结果。

---

## 0. 本计划解决什么，不解决什么

### 0.1 解决范围
- 新增并落地 `bipartite_align_transformer` scorer（soft alignment + 3 层 set-transformer 风格）。
- 训练默认切到 listwise + 新 scorer，推理默认切到新 scorer。
- 新建一组 runtime v2 脚本，做到“直接跑、不手改老脚本”。
- 增加数据链路体检脚本，优先发现缓存/样本/口径错位。
- 评测固定输出 `Recall@k / Recall@P / CoarseRankRecall@k / CoarseRankRecall@P / Gain@k / Gain@P`。

### 0.2 明确不做（当前阶段）
- 不做大规模性能压测和复杂并行调参。
- 不做全量 ablation matrix 扫参。
- 不做多模型混训和复杂蒸馏。
- 不在第一轮引入过多花哨增强（例如多分支对称对齐 + learned pooling 同时上线）。

---

## 1. 以往经验教训（本次强制吸取）

1. **缓存污染是最高优先级风险**  
`users` 与 `users_smoke` 必须硬隔离；禁止 smoke 写正式缓存。

2. **无签名模式不能再做“expected 行数强绑”**  
存在缓存即复用，采样发生在评测阶段，不允许因条数不等重建正式缓存。

3. **默认值必须可解释，且贴近正式流程**  
`max-queries` 默认全量（0）；避免隐式 2000 截断造成训练错觉。

4. **pairwise/listwise 语义要在脚本层明确**  
默认链路以 listwise 为主，pairwise 保留为回归对照，不再混淆。

5. **评测必须能区分“模型贡献”与“融合遮蔽”**  
必须同时看 `S-only` 和 `mix(auto)`，否则新 scorer 真实增益会被融合通道掩盖。

6. **不要用超长读取命令折腾终端**  
读取大文件只看必要片段（`rg` + `TotalCount` + 精确定位），避免卡顿。

## 2. 新主链路设计原则（B 方案项目化改写）

### 2.1 模型原则
- 输入保持 `q_vecs/m_vecs` token 级向量组（当前上限 8）。
- 显式构建 `8x8` 相似矩阵并做 soft alignment。
- 保留 token 细节到 transformer 前，不提前压成低维统计。
- 输出 raw score（logit），不做 sigmoid 压扁。

### 2.2 工程原则
- 新旧并行一段时间，但**新链路默认**。
- 所有默认入口脚本都直接指向新 scorer 与新权重路径。
- 旧链路改名为 `legacy` 路径，避免误用。
- 任何“可能覆盖正式缓存/权重”的动作都先写备份。

## P1（模型接入期，新增 scorer 但不破坏旧逻辑）

### P1.1 新 scorer 模块
- 新增：`src/memory_indexer/learned_scorer_bipartite.py`
- 包含：
  - `BipartiteAlignTransformer(nn.Module)`
  - `BipartiteLearnedFieldScorer`（实现 `score` 与 `score_many`）
- 第一版结构：
  - shared projection: 384 -> 192（可配）
  - `S = q @ m^T / tau`
  - `A = softmax(S, -1)`, `m_aligned = A @ m`
  - token feature concat: `[q, m_aligned, q-m_aligned, q*m_aligned]`
  - 3 层 encoder（无位置编码，Pre-LN）
  - mean pooling + MLP -> raw score
- 必做：
  - padding + mask 支持（不足 8 的样本）
  - batch 推理路径（`score_many`）保证可用

### P1.2 scorer 装配与元信息
- 调整：`src/memory_indexer/scorer.py`
  - 支持根据权重 `meta.model_family` 自动加载 tiny 或 bipartite scorer。
- 权重 meta 必含：
  - `model_family`
  - `loss_type`
  - `dataset`
  - `encoder_id`
  - `vectorizer_strategy/k`

### P1 验收标准
- 不改训练代码也可加载新 scorer 权重做推理。
- `eval_router` 在 `S-only` 下可稳定跑通（无 shape 错误、无 mask 误算）。

## P2（训练协议期，listwise 主链路统一）

### P2.1 训练入口升级
- 调整：`scripts/memory_indexer/train_reranker.py`
  - 新增参数：`--model-family tiny|bipartite`（默认 bipartite）
  - 默认 loss：listwise
  - 保持 negatives：粗召回非正例优先，候选不足再全库补
  - 保持 cache 命中跳过 HF/E5 初始化逻辑

### P2.2 置换增强（可控上线）
- 新增参数（默认先关，第二轮再开）：
  - `--permute-q-tokens`
  - `--permute-m-tokens`
- 仅当第一轮指标确认有效后再开启，避免一次引入过多变量。

### P2 验收标准
- 30 epochs 训练日志可见稳定收敛。
- `Recall@P`、`Gain@k/Gain@P` 至少有一项较 tiny 基线显著改善（在同评测口径下）。

## P4（评测与记录期，防“loss 好看但指标不涨”复发）

### P4.1 固定对照组
- 每次正式评测固定输出两组：
  - `S-only`（看模型真实排序能力）
  - `mix(auto)`（看线上融合表现）

### P4.2 固定指标面板
- 必报：
  - `Recall@k`
  - `Recall@P`
  - `CoarseRankRecall@k`
  - `CoarseRankRecall@P`
  - `Gain@k`
  - `Gain@P`
  - `MRR`, `Top1`

### P4.3 训练日志强制字段
- 每轮输出：
  - `rank_loss`
  - （如启用）`listwise_hard_loss`
  - 训练样本规模、平均 positives、负例来源占比

## 7. 回滚与止损策略

- 权重回滚：每次覆盖正式权重前自动备份 `*.old_<timestamp>.pt`。
- 缓存止损：正式 alias 下禁止 smoke 参数写入。
- 脚本回滚：保留 legacy 脚本，runtime_v2 故障时可一键切回 legacy。

---

## 8. 本计划的成败判断

若满足以下条件，即判定阶段成功：
- 新链路成为默认入口，用户无需改老脚本参数即可直接跑。
- 同口径评测下，`S-only` 指标较 tiny 版本有明确提升。
- 未再出现“缓存误重建、alias 污染、默认截断、pairwise/listwise 混线”问题。
- 开发日志可追溯每次权重来源、缓存命中状态和关键指标变化。
