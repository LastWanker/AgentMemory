V5 联想通路全量提纲（待批阅）

一、项目定位

V5 不再只是一个 coarse-only 的精确检索 demo。

V5 最终要形成两条并列通路：

1. 精确通路
负责把“用户明确在找什么”找回来。

2. 联想通路
负责把“这句话会激活什么相关记忆”找出来。

两条通路默认并列输出，不做融合打分，不做 reranker，不把联想结果硬塞回 coarse 排序里。

最终给上层 LLM 的输入可以是：

1. coarse topK
2. associative topM

并且从 associative 里剔除 coarse 已命中的重复项。

这不是“加一个花里胡哨的附属功能”，而是 V5 的第二主功能。

二、总目标

一步到位地做出一条能工作的联想检索链：

1. 原句直接进入 LLM
2. LLM 从原句中提纯概念
3. 构建概念层级图
4. query 命中概念种子
5. 概念传播点亮
6. 从亮概念取回 memory
7. 形成联想候选结果

目标不是学术完美，而是直接做成一个能展示、能解释、像作品的 demo 系统。

三、核心世界观

1. L0 就是原句本身

L0 不是什么“10 个脏词层”，也不是一层要长期保留的图节点库。

L0 就是：

- 原始句子
- 原始段落
- 原始脏文本

它可以很脏，可以很碎，可以直接整句喂给 LLM。

L0 的作用只有三个：

- 给 LLM 看原始证据
- 给概念抽取提供上下文
- 给展示保留人类可读来源

L0 不进入联想检索主传播链。

2. 真正的联想链发生在概念层

主传播层是：

- L1：具体概念
- L2：分类概念
- L3：领域概念

query 不是直接在 memory 上做第二次粗召回。

query 先命中 L1 种子，再把概念区域点亮，再从点亮的概念区域中取回 memory。

3. 联想感来自“先亮区域，再亮内容”

如果只是 query->memory 排序，那只是另一种检索。

如果是 query->concept region->memory，就更接近 activation retrieval。

四、层级设计

1. L0：原句层

- 原始句子
- 原始记忆文本
- 不清洗过度【对LLM来说，不太需要清洗吧。ps：你说的保留极小清洗（超长乱码/超长代码块截断）我同意。】
- 不做主传播

2. L1：具体概念层

这是整个系统最关键的一层。

L1 应该长得像：【不不，这里我要着重批阅一下，因为这肯定涉及到提示词的设计】

- 黄山无人机罚款争议【这个不好，黄山，无人机，罚款，争议还是要分开的。】
- 微积分应用题【√，方便往上走变成数学知识，数学题】
- 线代推导【√，但最好是线性代数】
- 国际政治战争讨论【不太好，至少具体到国家，比如巴以冲突】
- DeepSeek API 接入【这种建议拆词，而且主攻entity，比如Deepseek API，接入就不要了。】
- 聊天记忆切分【可以。】

L1 必须足够具体，足够像“某一类记忆片段会挂上去的概念”。

3. L2：分类层

L2 比 L1 抽象一些，但不能太空。

例如：【你举的例子不好。从这一步开始，就要偏词组或词了。而且，要相当接地气生活化的描述。用户只会说：我有个数学题让你看看。而不会说我遭受了数学学习问题。】

- 数学学习问题【数学题】【大学数学】
- 编程工程问题【编程】【代码】
- 国际时政讨论【国际时政】
- 聊天记忆工程【记忆算法】

4. L3：领域层

L3 是最顶层的主题或领域。

例如：

- 学习
- 编程【科技最好】
- 时政
- 生活

L3 不能空到“世界”“知识”“事情”这种废词层。【如果出现了那就出现了，提示词约束即可。也许用户就是喜欢聊宇宙世界呢。】

五、概念命名规则

1. LLM 负责生成 canonical concept name

概念命名必须可独立理解。【√】

能带定语时就带定语。【明显的一词多义词，则能带定语就带定语】

能组合成明确短语时就组合成明确短语。【能短则短，不能短再组合】

2. 不强制“歧义词一定不能独立存在”

例如“苹果”这种词，不要求一律只能当 alias。

如果在数据里它稳定指向“水果”语义，那它就可以直接成为水果概念节点。

换句话说：

不是先验禁止“苹果”这种裸词，
而是根据实际语义归属决定它最后会被聚到哪里。

3. 真正要避免的是错误 sense 混合

如果“苹果”在某批数据里既有水果又有公司又有手机，
那就必须通过上游命名让 LLM 产出更明确的概念：

- 苹果水果
- 苹果公司
- 苹果手机

但如果当前语料里它本来就只有水果义，
就没必要硬加戏把它改成更长词。

六、同义、同名、歧义处理

1. 同义词处理

同义词可以被合并到一个 canonical node。
【总归还是通过词向量相似度判断！比如0.85】【但是未必以后没有什么同义词词典之类的】
例如：

- 大模型接口
- LLM API
- 大模型 API

最终可以归到同一个概念节点。

2. 同名不同义处理

这是重点。

不能靠 embedding 自动兜底。

也不能指望聚类后自然长对。

必须在 LLM 概念抽取阶段就要求：

- 遇到歧义词时主动补限定语
- 遇到同名不同义时拆成不同概念

也就是说：

LLM 先做命名消歧
E5 再做向量合并和近邻归并

3. 当前明确不做 secondary_parent_ids

第一版不保存什么 secondary parent。

也不保存一堆“备选父节点”。

因为你要的是一步到位，不是给未来版本埋一堆可选复杂度。

七、层级关系设计

1. 主层级关系是父子关系

L1 -> L2 -> L3

每个节点在主层级上只有一个父节点。

这样做的原因不是“世界天然如此”，而是：

- 传播逻辑最稳定
- 工程实现最清晰
- 展示解释最好说

2. 多父不是当前主设计

“一个节点有多个父亲”这件事，当前不作为系统主逻辑。

如果一个概念看起来应该挂多个父类，优先反思两件事：

- 这个概念是不是没命名清楚
- 这个概念是不是本来就该拆成两个概念

也就是说：

先靠消歧和拆分解决，
而不是一开始就让传播图变成多父 DAG 乱图。

3. 苹果例子怎么处理

你的判断我接受：

单纯一个“苹果”，如果当前数据里它就是水果义，那它就应该被聚类成水果。

不要为了理论上的歧义把每个裸词都硬拆得过细。

所以规则应该是：

- 数据里没歧义，就正常聚
- 数据里出现歧义，就靠 LLM 加限定语拆开

八、图结构

系统需要的表很简单：

1. Memory 表

- memory_id
- raw_text
- optional meta

2. Concept 节点表

- node_id
- level
- canonical_name
- aliases
- vec

3. Parent 边表

- child_id
- parent_id
- conf

4. Memory 挂载表

- memory_id
- l1_node_id

5. L1 倒排

- l1_node_id -> memory_ids

九、离线构建总流程

这是一步到位版本的真正主线。

阶段 0：准备原始记忆

输入就是 V5 当前 memory 数据。

不做保守式过度清洗。

长文本可以适度裁掉极端噪声，但不做“先切得很细再喂 LLM”这种保守策略。

阶段 1：对每条 memory 直接做 LLM 概念提取

输入：

- memory 原句或原始文本

输出：

- 1~5 个 L1 概念【这里改一下，上限10个，中短文本推荐1~5个。面对长文本时候可以适量增加】

要求：

- 必须尽量具体
- 必须能独立理解
- 遇到歧义要主动消歧
- 不要抄原句整段
- 允许直接从脏原句里提纯

这一步是整个系统的根。

阶段 2：L1 概念归并

把所有 L1 候选 embedding 化。

对已有 L1 概念库找近邻：

- 足够接近就合并【0.85才合并，但这是个比较高的值。】
- 不够接近就新增

得到最终的 L1 canonical node 集合。

阶段 3：memory 挂载到 L1

每条 memory 可以挂多个 L1。

这是联想通路的关键：

记忆不是直接挂高层，而是先强挂在具体概念层。

阶段 4：为每个 L1 生成 L2/L3 候选【还是分两次吧。每次都有不同的规则和参考。L1->L2,L2->L3.而且，都是成组的，意味着一次把一个记忆的1~10的L1给LLM。】

仍然使用 LLM。

输入：

- L1 canonical name
- 必要时带少量来源句子【不必要。因为把概念成组给LLM了】

输出：

- 该 L1 所属的 L2
- 该 L1 所属的 L3【还是分两次吧。每次都有不同的规则和参考。】

注意：

这里的 L2/L3 也是先生成概念节点，
再统一做去重和归并，
不是让 LLM 直接决定最终全图结构。

阶段 5：父子边确定
【聚类可能可以利用sklearn的轮子：sklearn的KMeans（聚类，且允许指定初始中心）如果太慢，也可以看看MiniBatchKMeans】
对于每个 L1，找一个最合适的 L2 父节点。【不应该是一个一个地】

对于每个 L2，找一个最合适的 L3 父节点。【不应该是一个一个地】

边的确定方式可以是：【这里有一点问题：一个一个来不像是聚类像是分类。】
【我更想要那种半监督聚类，种子聚类（Seeded Clustering），这总归和一个一个分类不一样吧。】
- 先看 LLM 给出的归属【no，直接完全交给有凝结核的聚类(×)。ps：聚类决定结构，LLM只负责名门，合理。我说的凝结核就是上层的命名，所以不冲突。】
- 再看 embedding 相似度
- 最终选一个最合理父节点

如果用 KMeans / MiniBatchKMeans + 凝结核初始化（seed center）：

- “父节点”就是样本被分配到的中心（cluster center 对应父概念）
- 父子边分数可直接来自“子向量 vs 父中心向量”的相似度（例如 cosine）
- 或使用距离转分数（如 1 - normalized_distance）

这样可同时得到：聚类归属（父子结构）+ 可传播的父子边置信度分数。

重点不是形式绝对纯，
重点是最后得到一棵稳定可传播的主层级图。

阶段 6：构建联想桥接边

这是让系统更像“联想”的关键。

除了父子层级边，还要额外构建 L1-L1 稀疏桥接边。

桥接边来源可以是：

- 高频共现
- 同 cluster 共现
- 同一记忆共挂接
- 近邻概念共现

例如：

- 黄山无人机罚款争议 <-> 消防员
- 微积分应用题 <-> 线代推导
- 聊天记忆切分 <-> 对话聚类

桥接边不是主层级边。

桥接边只负责增加联想味，不负责定义父子归属。

桥接边构建建议与 KMeans 主层级解耦：

- KMeans 负责主层级找父（L1->L2->L3）
- 桥接边单独走图方法（语义近邻 + 共现权重）再稀疏化
- 不用 KMeans 的簇归属直接当桥边，否则会削弱跨簇联想

十、在线检索总流程

阶段 1：query 注入

当前第一版，在线注入明确走 dense 主线，不走 LLM 在线提纯。

query 原句直接进入 E5 检索：

1. 用 query 直接去打 L1/L2/L3 三层概念库
2. 三层结果不分层保底，不做人为按层配额【这里冻结：L1/L2/L3 公平竞争，听天由命】
3. 谁分高谁就进入初始 seed 集合

也就是说：

- 注入点不再限定只能是 L1
- 注入点可以直接落到 L1 / L2 / L3 任意层
- 不做“每层各取一批 seed”的分层取样策略

补充规则：

1. 当前暂时不加“整句 + 标点切分片段”这种多路注入逻辑
2. 当前先只用完整 query 做 dense 注入
3. 可以保留一个很弱的字符串 boost：
   - 如果 query 中直接出现某个 canonical concept name
   - 则给它一个额外 boost
   - 但最多只允许这种字符串命中额外占 1 个 seed 名额
4. 在线注入主线仍然是 dense，不让字符串匹配反客为主

关于方向：

1. 当前主分先用正常方向：
   - query 当 query
   - concept 当 passage
2. 反向打分（concept 当 query，query 当 passage）如果要试，只作为后续小补分，不作为第一版主设计

阶段 2：概念点亮

这里的核心，不是“找一条路径”，而是“让 query 激活一个概念区域”。

所以第一版主算法定义为：

- 受控点亮
- 带状态传播
- 不是最短路搜索

注入后的 seed 节点，统一转成 activation packet 往图里传播。

每个 packet 至少包含：

- node_id
- level
- score
- ttl
- came_from
- origin_type

其中：

1. direct
表示这个节点是 query 直接命中的 seed

2. from_child
表示这个节点是被自己的儿子点亮的

3. from_parent
表示这个节点是被自己的父亲点亮的

第一版固定规则：

1. 每个注入点初始 ttl = 2
2. 上溯不消耗 ttl
3. 下溯消耗 ttl
4. L1 的“下溯”不是树边，而是走 L1-L1 bridge
5. bridge 扩散也消耗 ttl

传播状态机：

1. direct
- 可以上溯
- 可以下溯
- 但不走回头路

2. from_child
- 可以继续上溯
- 也可以下溯
- 但不再下回刚刚点亮自己的那个儿子分支

3. from_parent
- 不再继续上溯
- 只允许继续下溯
- 也不走回头路

按层理解：

1. 如果直接命中的是 L3
- 不再上溯
- 直接按子孙边往下走
- 但下溯要消耗 ttl，所以从 L3 下来后通常不会再剩下 bridge 预算

2. 如果直接命中的是 L2
- 可以上溯 1 次到 L3
- 也可以下溯到 L1
- 但上去再下来不走回头路

3. 如果直接命中的是 L1
- 可以上溯 2 次到 L2 / L3
- 也可以做 bridge 扩散
- 如果 bridge 后还剩 ttl，则可以继续找 bridge 的 bridge

这一步的结果不是 memory 分数，
而是一张当前被激活的概念区域图。

阶段 2.5：下溯时的分支抽样

当前不建议把一个被点亮的父节点所有儿子全放开。

而是：

1. 根据 parent->child 的 conf 做权重分布
2. 先对 conf 做锐化，避免 0.94/0.95/0.96 看起来像 1:1:1
3. 再按这个分布做带固定 seed 的不放回抽样
4. 每次只抽一部分儿子继续下溯

也就是说：

- 下溯是受控展开
- 不是整棵子树洪泛
- 同一个 query 因为随机种子固定，应当得到稳定结果

阶段 2.6：点亮亮度更新

第一版亮度公式先保持简单可解释：

1. 上溯：
   parent += child_score * edge_conf * alpha_up

2. 下溯：
   child = max(child, parent_score * edge_conf * alpha_down)

3. bridge：
   peer = max(peer, src_score * bridge_weight * alpha_bridge)

重点先不在公式花哨，
而在：

- 状态机正确
- ttl 正确
- 不走回头路
- 下溯受控展开

阶段 2.7：点亮后的可解释性

主算法不先做寻路，
但要保留每个 node 被点亮时的来源痕迹，
这样后续可以回放：

- 是直接 seed 命中
- 还是从儿子上来的
- 还是从父亲下来的
- 还是经过 bridge 横跳过来的

阶段 3：从亮 L1 回收 memory

取最亮的一批 L1，
通过倒排拿回 memory 候选。

注意：

memory 仍然只从 L1 回收，
不直接从 L2/L3 回收。

因为 memory 的主挂载层就是 L1，
L2/L3 只负责组织和传播。

memory 的联想分数，不建议简单按命中的 L1 亮度求和。

因为：

- 不同 memory 挂载到的 L1 数量不一样
- 简单求和会偏向“挂得多的记忆”

第一版更适合做饱和式聚合，例如 noisy-or 思路：

1. 每个命中的亮 L1 给 memory 一个局部命中概率
2. 多个命中共同抬高最终分数
3. 但不会像简单求和那样无限线性叠加

还可以额外保留少量补项：

- 是否命中过 direct seed 路径
- 是否命中过 bridge 路径

这样可以区分：

- 只是同类命中
- 还是确实发生了联想横跳

阶段 4：结果清理

做基础去重：

- 与 coarse 重复的剔除
- 同一小片段刷屏的压制

最后形成 associative topM。

十一、联想点亮机制

这是这条通路最重要的算法心脏。

点亮来源有三部分：

1. seed 注入亮度
来自 query 对概念图的直接 dense 命中

2. 层级传播亮度
来自 L1/L2/L3 上下传播

3. 桥接扩散亮度
来自 L1-L1 的联想桥接

一个更像联想的系统，不是只有层级传播，
还必须有桥接扩散。

否则只会像分类系统，不像联想系统。

补充冻结口径：

1. 第一版在线点亮不走 LLM API 提纯概念
2. 第一版在线主注入走 dense
3. L1/L2/L3 注入点不做分层保底或分层配额
4. 每个注入点 ttl=2
5. 上溯不扣 ttl，下溯与 bridge 扣 ttl
6. 被父亲点亮的节点，不再继续上溯
7. 被儿子点亮的节点，可以继续上溯和下溯，但不走回头路
8. L1 的下溯就是 bridge 扩散
9. 主算法先定义为“受控点亮”，不是“找最短路”

十二、为什么桥接边必须做【认可！】

如果没有桥接边，系统会偏向：

- 同类归类
- 上下位传播
- taxonomy recall

但联想常常是：

- 事件里的共现对象
- 讨论里的常见搭配
- 同一记忆群里的伴随主题

这类关系不是父子关系，
必须单独建桥。

所以联想图 = 主层级树 + 稀疏桥接边。

十三、LLM 使用原则

这里不保守。

1. 原句直接给 LLM

不要先把句子剪得太瘦。

2. LLM 是主抽取器，不只是辅助润色

它要负责：

- 从脏原句中提纯概念
- 在歧义时补限定语
- 给出高质量 L1

3. 不拿“怕费 token”当核心约束

当前目标是做出作品，不是先把成本优化到死。

十四、E5 的角色

E5 不是主命名器。

E5 主要负责：

- 概念向量化
- 同义归并
- 近邻查找
- 在线 query 注入落点

也就是说：

- LLM 负责命名和消歧
- E5 负责空间组织和检索

十五、展示设计

最终 demo 展示必须一眼能看出两条通路不同。

建议界面或输出层明确分成：

1. 精确记忆
2. 联想记忆

并且联想记忆最好附带：

- 命中的种子概念
- 被点亮的高层概念
- 命中的桥接概念

这样联想结果才是可解释的。

十六、为什么这套方案像作品

因为它具备完整系统结构：

1. 有原始证据层
2. 有概念抽取协议
3. 有层级图
4. 有桥接图
5. 有点亮算法
6. 有双通路展示

这已经不是“给检索加个小按钮”，而是一套可以讲明白的 retrieval design。

十七、当前冻结结论

下面这些点当前先冻结，不再来回摇摆：

1. L0 就是原句本身
2. LLM 不保守，直接吃原句做概念提纯
3. 主传播层是 L1/L2/L3
4. 每个节点主层级只有一个父亲
5. 不做 secondary_parent_ids
6. 歧义优先靠 LLM 命名时加定语解决
7. 如果数据里本来没歧义，裸词可以正常存在并被聚对
8. 联想图必须包含 L1-L1 稀疏桥接边
9. 联想通路独立排序，不依赖 coarse 分数
10. 最终输出是 coarse + associative 双列表
11. 在线注入第一版不走 LLM API
12. 在线注入主线是 dense
13. 在线注入点允许直接命中 L1/L2/L3，三层公平竞争
14. 不做人为“按层各取 seed”配额
15. 字符串 boost 只做弱补分，且最多额外占 1 个 seed 名额
16. 每个 seed packet 的 ttl = 2
17. 上溯不扣 ttl，下溯和 bridge 扣 ttl
18. 被父亲点亮的节点不再上溯，只继续下溯
19. memory 只从最终亮 L1 回收，不直接从 L2/L3 回收

十八、下一步工作指令

你批阅这份提纲后，就直接按这份提纲开始干。

后续实现顺序建议固定为：

1. 定义概念节点和边的数据契约
2. 先打通 LLM 提取 L1
3. 再打通 L1 归并和 memory 挂载
4. 再做 L2/L3 和主层级父子边
5. 再做 L1-L1 桥接边
6. 再做在线点亮和 memory 回收
7. 最后接 chat/demo 展示

不要中途退回去做“保守缩小版”。

不要把它又做成 reranker。

不要把联想结果依附在 coarse 上。

这份提纲的目标，就是让后续实现不走偏。
