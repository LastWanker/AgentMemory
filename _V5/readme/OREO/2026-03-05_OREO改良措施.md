# OREO 改良措施（2026-03-05）

## 1. 目标

在不破坏当前“串行后处理、可热关闭、不干扰主链路”的前提下，降低两难：

- 保守参数几乎不触发
- 放松参数随机扰动过高

## 2. 核心判断

主要问题不是“要不要 Oreo”，而是三件事：

1. 训练任务耦合：`unrelated` 与 `toforget` 共头共阈值
2. 线上融合粗糙：跨 type 直接取最大 suppress
3. 标定缺失：分布压缩在低分窄区间，门控阈值极敏感

## 3. 分阶段措施

## 3.1 阶段 A（先做，不重训）

### A1. 融合策略从“max(type)”改为“冲突感知融合”

当前 runtime 已有 `per_type_scores`，可以直接改融合逻辑：

- 设 `s_u=score(unrelated)`，`s_f=score(toforget)`
- 仅当 `max(s_u,s_f)` 过阈且 `|s_u-s_f| >= min_type_gap` 才允许进入 suppress 候选
- 若两者都高且接近（类型不确定），默认 `no-op`（或只做极弱降权）

效果：

- 减少跨语义互相拉扯导致的误触发
- 不改训练即可先压住 random 扰动

### A2. 门控增加“候选相关性底线”

新增一层轻量安全门（不侵入主召回器）：

- 对候选的 `base_score/cosine` 设 `min_base_relevance`
- 低相关候选直接不进入 suppress 门控链

这相当于你说的“先把和 query 相似度很低的拦下来”，是额外保险丝，不替代现有 threshold/margin/delta/zscore。

### A3. 分 lane 参数固定化

- `coarse` 与 `association` 按各自分布单独阈值
- 至少拆成：
  - `threshold_coarse / threshold_association`
  - `min_delta_coarse / min_delta_association`
  - `min_zscore_coarse / min_zscore_association`

## 3.2 阶段 B（小规模重训，优先级最高）

### B1. 改成“共享底座 + 类型分头”

推荐架构：

- 共享：Oreo BlockA + Memory Layer
- 分头：`head_unrelated`、`head_toforget`

线上不再“max(type)”盲选，而是按融合策略决策。

### B2. 冲突 key 数据治理

对同 `(q,m,lane,feedback_type)` 的双标签冲突做仲裁：

- 时间优先（近反馈覆盖旧反馈）
- 或事件置信度优先
- 冲突样本打低权重（不是直接删除）

### B3. 标定层

训练后增加 per head/per lane 标定：

- 温度缩放（temperature scaling）
- 阈值搜索以 two_cohort 指标为目标，而不是固定 0.5

## 3.3 阶段 C（评测口径升级）

two_cohort 增加以下必看项：

1. `cross_type_conflict_trigger_rate`  
   类型冲突样本被触发 suppress 的比例（应显著下降）
2. `ambiguous_case_noop_rate`  
   类型不确定样本被保守跳过的比例（应上升）
3. 分 lane 的 random 稳定性  
   coarse 与 association 分开看，不再只看合并指标

## 4. 实施顺序（建议）

1. 先落 A1/A2（无重训，可立即验证）
2. 再做 B1（类型分头）+ B3（标定）
3. 最后做 B2（冲突治理）并回跑 two_cohort 全量

## 5. 验收门槛（建议）

至少满足：

1. `SUPPRESSOR_ENABLED=0` 完全等价当前主链路
2. random cohort：
   - `top1_changed_rate <= 0.15`
   - `top5_set_changed_rate <= 0.20`
3. feedback cohort：
   - `feedback_pair_removed_rate` 相对当前基线有实质提升
4. `cross_type_conflict_trigger_rate` 明显低于当前实现

## 6. 本文对应的工程约束

所有改造都保持：

- suppressor 作为后处理串行层
- 可热关闭
- artifact 缺失时 graceful no-op
- 不改动 coarse/association 主召回内部逻辑
