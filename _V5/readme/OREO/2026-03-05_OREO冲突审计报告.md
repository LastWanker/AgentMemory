# OREO 冲突审计报告（2026-03-05）

## 1. 审计范围

- 数据集：`data/V5/suppressor_oreo_r1_dataset`
- 模型产物：`data/V5/suppressor_oreo_r1_artifact`
- 审计结果：`data/V5/suppressor_oreo_r1_eval/conflict_audit_report.json`
- 审计脚本：`_V5/scripts/suppressor/audit_oreo_conflicts.py`

本报告回答三个问题：

1. `unrelated` 与 `toforget` 是否存在语义冲突
2. `## 7 unrelated 样本构造` 与 `## 8 toforget 样本构造` 是否“没道理”
3. 当前 Oreo 头为何出现“保守时全不动、放松时乱动”二难

## 2. 关键结论（先给结论）

1. 夹心（Oreo）结构本身没有被证伪，问题主要在“多语义混训 + 标定 + 融合策略”。
2. `##7/##8` 的样本构造思路本身有道理，但合并到单头单阈值后确实产生冲突和干扰。
3. 当前模型输出分布高度压缩在低分窄区间（约 `0.14~0.17`），导致默认门控几乎全不触发。
4. 放松门控后触发变多，但由于类型冲突和融合过于激进，random cohort 扰动迅速放大。

## 4. 输出分布与校准问题

模型得分分布（按训练口径构造特征后直接前向）：

- 全体均值：`0.1574`，中位 `q50=0.1554`
- 正样本均值：`0.1612`
- 负样本均值：`0.1571`

正负均值只差约 `0.0042`，分位数高度重叠。  
这意味着：

1. 0.5 阈值下几乎必然“全负”（与训练日志里的 `pos_recall=0` 一致）
2. 一旦把阈值压低到 `0.15` 附近，触发量会突然跃迁并带来大扰动

对应本次 two_cohort 观测：

- 保守网格（`threshold>=0.5`）几乎全 0 变化
- 松门控（`threshold=0.15`）出现 `feedback_removed≈12.37%`，但 `random_top1≈45.5%`，`random_top5_set≈80%`

## 5. 对 `##7` / `##8` 的判断

结论不是“没道理”，而是“单模型单融合下会互相干扰”：

1. `unrelated` 与 `toforget` 各自语义是成立的
2. 但两者在部分 `(q,m,lane)` 上会给出不同甚至相反训练信号
3. 若线上用“跨类型取最大 suppress”并共用同一阈值，冲突被放大

因此当前现象更像“任务耦合方式不对”，不是语义定义本身错误。

## 6. 本报告最终结论

1. Oreo 夹心结构是可行方向，但当前瓶颈在任务拆分与校准。
2. 样本构造原则可保留，不建议推翻 `##7/##8`。
3. 下阶段应优先处理：
   - 类型冲突融合策略
   - lane/type 分阈值与校准
   - 冲突 key 的数据治理
