# V5 点亮算法实现草案

更新时间：2026-03-03

## 1. 目标

这份文档不是讨论稿，而是实现稿。

目标是把 V5 联想通路中的“点亮算法”收成一个可直接编码的第一版方案。

当前明确不做：

1. 不做在线 LLM 提纯注入
2. 不做最短路/寻路作为主算法
3. 不做按层配额的 seed 采样
4. 不做从 L2/L3 直接回收 memory

当前明确要做：

1. dense 主注入
2. L1/L2/L3 三层公平竞争
3. 带状态的受控点亮传播
4. L1-L1 bridge 扩散
5. 从最终亮 L1 回收 memory
6. 输出足够详细的 trace 方便 debug 和展示

## 2. 主世界观

主算法定义为：

`query -> seed 注入 -> 概念图受控点亮 -> 亮 L1 回收 memory -> associative results`

这里的重点不是“在图里找一条路径”，而是“让 query 点亮一个概念区域”。

路径不是主算法，只是解释产物。

也就是说：

1. 主输出是亮区，不是最短路
2. memory 分数来自亮区，而不是来自一条路径的路径分数
3. 路径回放只用于解释“为什么这个节点/记忆被点亮”

## 3. 输入与输出

### 3.1 输入

在线点亮算法输入：

1. `query_text: str`
2. `concepts.jsonl`
3. `parent_edges.jsonl`
4. `bridge_edges.jsonl`
5. `l1_matrix.npy`
6. `concept_matrix.npy` 或等价分层向量资产
7. `memory_links.jsonl`

### 3.2 输出

算法输出至少包含：

1. `hits`
联想 memory 结果列表

2. `trace`
完整点亮过程摘要

3. `activation_summary`
本次被点亮的 L1/L2/L3 概念摘要

4. `memory_score_breakdown`
每条命中 memory 的分数组成

## 4. 图资产约定

### 4.1 节点层

节点分三层：

1. `L1`
具体概念层，也是 memory 主挂载层

2. `L2`
分类层

3. `L3`
领域层

### 4.2 边层

边分两类：

1. `parent edge`
主层级边，只有：

- `L1 -> L2`
- `L2 -> L3`

2. `bridge edge`
水平桥接边，当前只允许：

- `L1 <-> L1`

### 4.3 memory 挂载

memory 只挂在 `L1`。

所以：

1. `L2/L3` 负责组织和传播
2. `L1` 负责最终回收 memory

## 5. 在线注入设计

## 5.1 注入主线

第一版在线注入只有一条主线：

`dense injection`

即：

1. 用用户完整 query 编码
2. 直接去打 L1/L2/L3 三层概念库
3. 把三层结果混在一起统一竞争

### 5.2 不做的事

当前不做：

1. 在线 LLM 注入
2. query 自动切成标点片段后多路注入
3. 人工规定“L1 取多少、L2 取多少、L3 取多少”

### 5.3 弱字符串 boost

可以加一个弱规则补分：

如果 query 字面上直接包含某个 canonical concept name，则：

1. 给该 concept 一个额外 boost
2. 但这种 boost 最多只允许额外占 1 个 seed 名额
3. 不能把字符串匹配变成主注入逻辑

### 5.4 注入公平竞争

L1/L2/L3 三层 seed 统一竞争，不做分层保底。

原因：

1. 用户可能直接问抽象层主题，如“社会与民生”
2. 如果强行只让 L1 主导，会损伤高层概念直击能力
3. 当前目标是让“落点听天由命”，谁更匹配谁进 seed

## 6. Seed Packet 数据结构

在线传播不直接传“节点亮了”，而是传“点亮包”。

建议数据结构：

```python
@dataclass
class ActivationPacket:
    packet_id: str
    node_id: str
    level: str
    score: float
    ttl: int
    source_kind: str
    origin_type: str
    came_from_node_id: str | None
    blocked_node_id: str | None
    hop_down_count: int
```

字段含义：

1. `packet_id`
本次包的唯一 id，便于 trace

2. `node_id`
当前所在节点

3. `level`
当前节点层级，取值 `L1/L2/L3`

4. `score`
当前包携带的亮度

5. `ttl`
剩余下溯预算

6. `source_kind`
当前包最初来自哪种注入：

- `dense`
- `string_boost`

7. `origin_type`
当前这个节点是如何被点亮的：

- `direct`
- `from_child`
- `from_parent`
- `from_bridge`

8. `came_from_node_id`
上一步从哪个节点传播而来

9. `blocked_node_id`
当前节点向下扩散时，不能回走的那个节点

10. `hop_down_count`
已经发生了多少次向下传播

## 7. Seed 生成流程

建议流程：

### 7.1 计算 dense 分数

1. 用完整 query 编码为 query vector
2. 分别与 L1/L2/L3 三层 concept matrix 做相似度计算
3. 各层拿到局部候选
4. 三层候选汇总到同一个候选池

### 7.2 统一排序

1. 将三层候选按统一分数排序
2. 加上弱字符串 boost
3. 选出总 seed topK

这里的 topK 是全局 topK，不是分层 topK。

### 7.3 生成初始 packet

每个 seed 生成一个初始 packet：

```python
ActivationPacket(
    packet_id=...,
    node_id=seed.node_id,
    level=seed.level,
    score=seed.score,
    ttl=2,
    source_kind=seed.source_kind,
    origin_type="direct",
    came_from_node_id=None,
    blocked_node_id=None,
    hop_down_count=0,
)
```

## 8. TTL 规则

这是第一版最重要的冻结口径之一。

### 8.1 基本规则

1. 每个 seed 初始 `ttl = 2`
2. 上溯不扣 ttl
3. 下溯扣 1 ttl
4. bridge 扩散也扣 1 ttl

### 8.2 含义

这表示：

1. 系统允许“理解更高层主题”而不把预算浪费在上溯上
2. 真正昂贵的是“向下展开”和“横向发散”
3. bridge 本质上应视为一种特殊下溯成本

### 8.3 直接效果

1. 从 L3 直接往下走两层后，通常没有剩余 bridge 预算
2. 从 L2 走到 L1 后，通常还能保留有限横跳能力
3. 从 L1 出发，可以横跳到 bridge 的 bridge，但深度有限

## 9. 传播状态机

### 9.1 状态类型

当前只保留四种 `origin_type`：

1. `direct`
2. `from_child`
3. `from_parent`
4. `from_bridge`

### 9.2 规则总表

#### A. `direct`

表示该节点是 query 直接命中的 seed。

允许：

1. 上溯
2. 下溯
3. 不走回头路

#### B. `from_child`

表示该节点是被自己的儿子点亮的。

允许：

1. 继续上溯
2. 继续下溯
3. 但下溯时不再走回点亮自己的那个儿子

#### C. `from_parent`

表示该节点是被自己的父亲点亮的。

允许：

1. 只继续下溯
2. 不再继续上溯
3. 不走回头路

#### D. `from_bridge`

表示该节点是经由 L1 bridge 横跳被点亮的。

允许：

1. 继续 bridge 扩散，只要 ttl > 0
2. 也可以上溯到父节点
3. 但不立刻走回刚刚那条 bridge

### 9.3 为什么这样定义

这样定义的目的：

1. 防止来回震荡
2. 防止一条边无限往返刷亮度
3. 允许“由下往上理解，再由上往下展开”
4. 允许 L1 做有限横跳

## 10. 传播动作定义

传播动作分三类：

1. `ascend`
2. `descend`
3. `bridge`

### 10.1 ascend

含义：

1. `L1 -> L2`
2. `L2 -> L3`

规则：

1. 不扣 ttl
2. 使用 parent edge 的 `conf`
3. 生成的新 packet：
   - `origin_type = from_child`
   - `came_from_node_id = child_id`
   - `blocked_node_id = child_id`

### 10.2 descend

含义：

1. `L3 -> L2`
2. `L2 -> L1`

规则：

1. 扣 1 ttl
2. 从候选儿子中抽样，而不是全放开
3. 生成的新 packet：
   - `origin_type = from_parent`
   - `came_from_node_id = parent_id`
   - `blocked_node_id = parent_id`

### 10.3 bridge

含义：

1. 仅限 `L1 -> L1`

规则：

1. 扣 1 ttl
2. 只能从 `L1` 发出
3. 生成的新 packet：
   - `origin_type = from_bridge`
   - `came_from_node_id = src_l1_id`
   - `blocked_node_id = src_l1_id`

## 11. 分支抽样策略

这是“受控点亮”是否像联想的关键。

### 11.1 不做洪泛

当一个高层节点被点亮时，不要把它所有儿子全放开。

否则：

1. 很容易整片子树一起亮
2. 图会更像 taxonomy recall，不像联想激活

### 11.2 权重构造

对候选儿子边的 `conf` 做锐化：

```python
w_i = exp(tau * (conf_i - max_conf))
```

其中：

1. `tau` 是锐化系数
2. `max_conf` 是该父节点所有候选儿子里最大的边置信度

### 11.3 抽样方式

1. 按 `w_i` 做不放回加权抽样
2. 随机种子固定为 `hash(query_text)`
3. 同一个 query 的结果应当稳定

### 11.4 抽样额度

第一版可用：

1. 抽取 50% 儿子
2. 但每次展开不超过固定上限

建议上限：

1. `L3 -> L2` 最多 5
2. `L2 -> L1` 最多 5
3. `L1 -> bridge` 最多 5

这只是第一版默认值，不是永久冻结真理。

## 12. 亮度更新

第一版公式保持简单可解释。

### 12.1 上溯

```python
parent_score += child_score * edge_conf * alpha_up
```

### 12.2 下溯

```python
child_score = max(child_score, parent_score * edge_conf * alpha_down)
```

### 12.3 bridge

```python
peer_score = max(peer_score, src_score * bridge_weight * alpha_bridge)
```

### 12.4 当前原则

第一版重点不在复杂公式，而在：

1. 状态机对
2. ttl 对
3. 不走回头路
4. 下溯抽样对
5. trace 能解释

## 13. 节点激活缓存

在线执行时，需要维护全局激活表。

建议结构：

```python
node_activation: dict[str, NodeActivationState]
```

其中：

```python
@dataclass
class NodeActivationState:
    node_id: str
    level: str
    best_score: float
    hit_count: int
    first_source_kind: str
    reasons: list[dict]
```

`reasons` 至少记录：

1. 来源 packet id
2. 来源节点
3. 来源类型
4. 动作类型
5. 使用的边权
6. 传播后得分

这样之后既能 debug，也能做解释回放。

## 14. memory 回收

### 14.1 回收入口

只从最终被点亮的一批 `L1` 回收 memory。

不允许：

1. 直接从 `L2` 回收
2. 直接从 `L3` 回收

### 14.2 候选 memory 生成

1. 取最终亮度最高的一批 `L1`
2. 用 `l1 -> memory_ids` 倒排拿回 memory
3. 聚合每条 memory 命中的多个 L1 贡献

### 14.3 不建议简单求和

原因：

1. memory 挂载的 L1 数量不一致
2. 简单求和会偏袒“挂点更多”的 memory

### 14.4 第一版推荐聚合

推荐饱和式聚合，如 `noisy-or`：

```python
p_i = clamp(lambda_ * l1_score_i, 0.0, 0.95)
memory_score = 1.0 - product(1.0 - p_i for p_i in hits)
```

这样：

1. 多命中会增强
2. 但不会无限线性叠加
3. 更适合“多个激活证据共同点亮同一条记忆”

### 14.5 可选补项

可以保留两个小 bonus：

1. `seed_bonus`
如果该 memory 命中了 direct seed 所在 L1

2. `bridge_bonus`
如果该 memory 命中了 bridge 扩散带来的 L1

这样能区分：

1. 普通同类命中
2. 确实发生横向联想命中

## 15. 去重与 coarse 剔除

联想链最终输出前，需要做基础清理。

### 15.1 coarse 剔除

并列输出时，联想结果应剔除 coarse 前几名已命中的 memory。

这是当前主口径之一。

### 15.2 刷屏压制

如果多个结果本质上来自同一小片段或同一局部区域，需要做轻压制，避免联想列表刷屏。

## 16. Trace 设计

这是实现时必须保留的部分，不是锦上添花。

### 16.1 顶层 trace

至少输出：

1. `query`
2. `seed_candidates`
3. `accepted_seeds`
4. `activation_counts`
5. `top_l1`
6. `top_l2`
7. `top_l3`
8. `bridge_hits`
9. `memory_score_breakdown`

### 16.2 seed trace

每个 seed 至少记录：

1. 命中节点 id
2. 命中层级
3. 原始 dense 分数
4. 是否吃到字符串 boost
5. 最终 seed 分数

### 16.3 传播 trace

每次传播至少记录：

1. from node
2. to node
3. action = ascend / descend / bridge
4. edge weight
5. old score
6. new score
7. ttl before / after
8. blocked node

### 16.4 memory trace

每条命中 memory 至少记录：

1. 命中了哪些 L1
2. 每个 L1 的亮度
3. noisy-or 聚合前局部值
4. 最终 memory_score
5. 是否吃到 seed_bonus / bridge_bonus

## 18. 第一版伪代码

```python
def associate(query_text: str) -> dict:
    seeds = build_dense_seeds(query_text)
    packets = init_packets(seeds, ttl=2)
    node_state = {}
    queue = packets[:]

    while queue:
        packet = queue.pop(0)
        accepted = merge_packet_into_node_state(packet, node_state)
        if not accepted:
            continue

        if can_ascend(packet):
            queue.extend(run_ascend(packet))

        if can_descend(packet):
            queue.extend(run_descend(packet))

        if can_bridge(packet):
            queue.extend(run_bridge(packet))

    bright_l1 = select_top_l1(node_state)
    memory_candidates = collect_memories_from_l1(bright_l1)
    memory_scores = score_memories(memory_candidates, bright_l1)
    memory_scores = remove_coarse_overlaps(memory_scores)
    hits = sort_and_trim(memory_scores)
    trace = build_trace(query_text, seeds, node_state, hits)
    return {"hits": hits, "trace": trace}
```

## 19. 当前冻结结论

本草案对应的当前冻结实现口径：

1. 在线主注入走 dense
2. 在线不走 LLM API 提纯
3. L1/L2/L3 seed 同池公平竞争
4. 不做分层配额 seed
5. 字符串 boost 最多额外占 1 个 seed 名额
6. seed packet 初始 ttl = 2
7. 上溯不扣 ttl
8. 下溯与 bridge 扣 ttl
9. `from_parent` 不再继续上溯
10. `L1` 的下溯就是 bridge
11. memory 只从亮 L1 回收
12. memory 分数第一版走饱和聚合，不走简单求和
13. 主算法是“受控点亮”，不是“最短路搜索”
